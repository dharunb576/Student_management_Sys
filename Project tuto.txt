## Line by Line Explanation (for app.module.ts)

### `ConfigModule.forRoot({ isGlobal: true })`
```
Loads your .env file
isGlobal: true → means every module in the app
can read .env values without importing ConfigModule again
```

### `TypeOrmModule.forRootAsync`
```
forRootAsync → means "wait for ConfigService to load first,
then set up the DB connection"
This is important because .env values must be
loaded before the DB tries to connect
```

### `config.get<string>('DB_HOST')`
```
Reads DB_HOST from your .env file
The <string> tells TypeScript what data type to expect
DB_PORT uses <number> because ports are numbers
```

### `entities: [__dirname + '/**/*.entity{.ts,.js}']`
```
This tells TypeORM to automatically find all your
entity files (your table definitions) anywhere in the project
Any file ending in .entity.ts will be picked up
```

### `synchronize: true`
```
In development → TypeORM auto-creates/updates
your DB tables based on your entity files

⚠️ WARNING: Never use synchronize: true in production
It can wipe or alter your real data
In production you use migrations instead
```

## How To Test The Connection

Run your NestJS app:
```
npm run start:dev
```

If connection is successful you will see:
```
[TypeORM] Connected to database ✓
[Nest] Application is running on: http://localhost:3000
```

If it fails you will see a database connection error — most common reasons:
```
❌ Wrong password in .env
❌ PostgreSQL service not running on your machine
❌ Database name doesn't exist yet in pgAdmin
```

### Full Flow in Simple Words
```
1. User registers     → role saved in database
2. User logs in       → role packed into JWT token
3. User sends request → token sent in header
4. JWT Strategy       → unpacks token, reads role
5. Roles Guard        → checks if role is allowed
6. Controller         → allows or blocks the action
```
